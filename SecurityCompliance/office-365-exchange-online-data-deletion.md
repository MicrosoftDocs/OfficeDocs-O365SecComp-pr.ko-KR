# <a name="exchange-online-data-deletion-in-office-365"></a>Office 365의에서 Exchange Online 데이터 삭제
내 Exchange Online에 다음과 같은 두가지 유형의 삭제: 소프트 삭제 및 하드 삭제 합니다. 이 사서함과 사서함 내의 항목을 모두에 적용 됩니다.

## <a name="soft-deleted-and-hard-deleted-mailboxes"></a>일시 삭제 하 고 하드 삭제 된 사서함
일시 삭제 된 사용자 사서함은 30 일 보다 작은 Azure Active Directory 휴지통에서 여전히 되었으므로 대 한 Office 365 관리 센터 또는 Remove-mailbox cmdlet을 사용 하 여 삭제 된 사서함입니다. 사서함에는 다음과 같은 방식으로 일시 삭제 될 수 있습니다.
- 사용자 사서함의 Azure Active Directory 사용자 계정이 일시 삭제 된 연결 된 (사용자 개체는 범위 또는 recycle bin 컨테이너에).
- 사용자 사서함의 관련된 Azure Active Directory 사용자 계정에 하드 삭제 되지만 소송 대기 아래에서 Exchange Online 사서함은 했거나 eDiscovery 보류 합니다.
- 사용자 사서함의 사용자 계정 지난 30 일; 내에서 제거 되었으며 Azure Active Directory에 연결 Exchange Online의 최대 보존 길이 얼마는 보관할 사서함 일시 삭제 된 상태에서 영구적으로 제거 하 고 복구할 수 없는 되기 전에 합니다.

사용자 하드 삭제 된 사서함은 다음 방법 중 하나에서 삭제 된 사서함이입니다.
- 사용자 사서함 일시 삭제 된 30 일 이상 되었으며 관련된 Azure Active Directory 사용자 하드 삭제 되었습니다. 예: 전자 메일, 연락처 및 파일에 있는 모든 사서함 콘텐츠를 영구적으로 삭제 됩니다.
- 사용자 사서함에 연결 된 사용자 계정을 Azure Active Directory에서 하드 삭제 되었습니다. 사용자 사서함은 이제 Exchange Online에서 일시 삭제 하 고 30 일 동안 일시 삭제 된 상태로 유지 합니다. 30 일 동안에서 새 Azure Active Directory 사용자를 동일한 **ExchangeGuid** 또는 **ArchiveGuid**, 원래 받는 사람 계정에서 동기화 및 Exchange Online에 대 한 새 계정을 어떠한 하는 경우 이렇게 하면의 하드 삭제 원래 사용자 사서함입니다. 예: 전자 메일, 연락처 및 파일에 있는 모든 사서함 콘텐츠를 영구적으로 삭제 됩니다.
- 일시 삭제 된 사서함은 **사서함 제거 PermanentlyDelete**를 사용 하 여 삭제 됩니다.

위의 삭제 시나리오는 사용자 사서함에 소송 보존 또는 eDiscovery 보류는 대기 상태에 있지 않은지 가정 합니다. 모든 유형의 사서함에서 대기 하는 경우에 사서함을 삭제할 수 없습니다. 모든 메일 사용자 받는 사람 유형에 대 한 모든 [보류](https://support.office.com/article/manage-legal-investigations-in-office-365-2e5fbe9f-ee4d-4178-8ff8-4356bc1b168e?ui=en-US&rs=en-US&ad=US) 설정을 무시 되 고 하드 삭제 또는 일시 삭제에 대 한 영향을 주지 않습니다.

## <a name="soft-deleted-and-hard-deleted-items"></a>일시 삭제 하 고 하드 삭제 된 항목
(예: 전자 메일 메시지, 연락처, 일정 약속 또는 작업)은 사서함 항목을 삭제 하는 사용자, 복구 가능한 항목 폴더 및 삭제 라는 하위 폴더에 항목이 이동 됩니다. 이것은 소프트 삭제 이라고 합니다. 삭제 된 항목 폴더는 사서함에 대 한 설정 하는 삭제 된 항목 보존 기간에 따라 달라 집니다 삭제에 저장 됩니다. Exchange Online 사서함 기본적으로 14 일에 대 한 삭제 된 항목을 유지 하지만 Exchange Online 관리자는 최대 30 일 까지의 기간을 늘리려면이 설정을 변경할 수 있습니다. (Exchange Online 사서함에 대 한 삭제 된 항목 보존 기간을 높이 자세한 단계에 대 한 참조 [Exchange Online 사서함에 대 한 변경을 얼마나 오래 영구적으로 삭제 된 항목 보관 됩니다](https://docs.microsoft.com/exchange/recipients-in-exchange-online/manage-user-mailboxes/change-deleted-item-retention).) 사용자에 복구 하거나 삭제 된 항목에 대 한 보존 기간이 만료 되기 전에 삭제 된 항목을 삭제할 수 있습니다. 이렇게 하려면 Microsoft Outlook 또는 웹에서 Outlook에서 지운 편지함 복구 기능을 사용 있습니다.

사용자가 Outlook 또는 웹에서 Outlook에서 지운 편지함 복구 기능을 사용 하 여 삭제 된 항목을 제거, 하는 경우이 이라고 하드 삭제 합니다. Exchange Online에서 단일 항목 복구가 사용 되며 기본적으로 새 사서함을 만들 때 하므로 관리자가 보관할 수 여전히 하드 지운 편지함 [복구](https://docs.microsoft.com/Exchange/recipients/user-mailboxes/recover-deleted-messages) 삭제 된 항목 보존 기간이 만료 되기 전에 합니다. 또한, 사용자 또는 프로세스에 의해 메시지를 변경 하는 경우 단일 항목 복구를 사용할 때 원본 항목의 복사본 유지도 됩니다.

## <a name="page-zeroing"></a>페이지 비우기
*Zeroing* 는 삭제 된 데이터를 복구 하는 것이 어려울 수 있도록 삭제 된 데이터를 통해에 오는 0 또는 이진 패턴을 기록 하는 보안 메커니즘입니다. Exchange Online에서 사서함 데이터베이스 자신의 단위의 저장소, *페이지* 를 사용 하 고 *페이지 비우기*를 호출 하는 덮어쓰기를 프로세스를 구현 합니다. 페이지 비우기은 기본적으로 사용 하도록 설정 하 고 고객에 의해 또는 Microsoft가 비활성화할 수 없습니다. 페이지 비우기 작업은 지정 된 데이터베이스의 모든 복사본이 페이지-비우는 유사한 방식으로 있도록 트랜잭션 로그 파일에 기록 됩니다. 페이지는 데이터베이스의 수동 복사본에 대해 0으로 설정 하기를 사용 하면 활성 데이터베이스 복사본에서 페이지 비우기 합니다.

페이지 비우기 하드 삭제 된 레코드를 통해 이진 패턴을 기록 합니다. 페이지 비우기 패턴 확장 가능한 저장소 엔진 (ESE) 작업 (내부 데이터베이스 엔진의 이름 사용 되는 서버에서 Exchange Online), 특정 되며 런타임 작업 시간대 백그라운드 데이터베이스 유지 관리 작업에 대 한 서로 다른 것입니다. (백그라운드 데이터베이스 유지 관리는 연속적으로 체크섬 및 검색 각 데이터베이스는 프로세스는 합니다. 주요 기능은 체크섬 데이터베이스 페이지에 이지만 공간을 정리 하 고 레코드 및 저장소 크래시로 인해 아웃 비워진 되지 않은 페이지 비우기도 처리 합니다.)

다음 표에는 특정 런타임 작업에 해당하는 채우기 패턴이 나와 있습니다.

| ESE 런타임 작업   | 채우기 패턴 |
|--------------------------|--------------|
| 바꾸기                  | R            |
| 레코드/Long 값 삭제 | D            |
| 비운 페이지 공간         | H            |


다음 표에는 ESE 백그라운드 데이터베이스 유지 관리 중에 발생하는 특정 작업에 해당하는 채우기 패턴이 나와 있습니다.

| ESE 백그라운드 데이터베이스 유지 관리 작업 | 채우기 패턴 |
|-----------------------------------------------|--------------|
| 레코드 삭제                                 | D            |
| Long 값 삭제                             | L            |
| 부분적으로 사용된 페이지의 비운 페이지 공간       | Z            |
| 사용되지 않은 페이지의 비운 페이지 공간               | U            |


### <a name="page-zeroing-process"></a>페이지 비우기 프로세스
페이지 비우기에 대 한 프로세스 삭제 시나리오에 따라 달라 집니다. 다음 표에서 데이터베이스 삭제 시나리오에 설명 하 고 페이지 비우기 기능이 발생 하는 경우.

| 데이터베이스 삭제 시나리오 | 데이터베이스 데이터를 비우기 위한 ESE 프로세스 및 시간대 |
|-----------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 항목 삭제 된 항목 보존 기간에 따라 만료 됩니다. | 비동기 스레드는 삭제된 데이터를 바이너리 패턴으로 덮어씁니다. 이 작업은 레코드 삭제 후 몇 밀리초 이내에 발생합니다. 비동기 비우기 작업이 여전히 해결되지 않은 상태에서 저장소 프로세스가 크래시되는 경우(또는 버전 저장소 증가로 인해 버전 저장소 정리가 취소되는 경우) 백그라운드 데이터베이스 유지 관리가 데이터베이스의 해당 섹션을 처리할 때 비우기가 완료됩니다. |
| 보기 시나리오: 웹 폴더 보기 (예: 대화 보기)에서 Outlook/Outlook에서 항목의 만료 | 백그라운드 데이터베이스 유지 관리가 데이터베이스의 해당 섹션을 처리할 때 데이터 비우기가 발생합니다. |
| 이동 사서함/사서함 삭제 시나리오: 원본 사서함 삭제 (휴지통의 삭제 된 사서함) | 백그라운드 데이터베이스 유지 관리가 데이터베이스의 해당 섹션을 처리할 때 데이터 비우기가 발생합니다. |

### <a name="mailbox-data-types-without-page-zeroing"></a>페이지 비우기 없는 사서함 데이터 형식
다음 사서함 데이터 형식 프로 비전이 없습니다 페이지 비우기에 대 한.
- **사서함 데이터베이스 트랜잭션 로그** -삭제 된 로그 파일을 저장 하는 파일 시스템에는 블록을 0으로 프로세스가 없는 일반 작업의 일부로 트랜잭션 로그를 삭제 하면 됩니다. 파일 시스템을 신속 하 게 다시 이용 하는 공간에 새로 만든된 로그 수 있지만 상황이 발생이 보장 되지 않습니다 가능성이 높습니다.
- **콘텐츠 인덱스 카탈로그 파일** -Exchange Online 검색 인덱싱 기능에 대 한 검색 Foundation (FAST로 알려져)를 사용 합니다. 검색 인덱스 카탈로그는 동일한 사서함 데이터베이스 파일 볼륨에 저장 된 여러 수십 파일의 구성 됩니다. 메시지가 사서함 데이터베이스에서 하드 삭제 되 면 검색 카탈로그에 연결 된 콘텐츠 즉시 삭제 되지 않습니다. 콘텐츠 삭제 발생 검색 Foundation 않는 경우 단일 더 큰 파일의 많은 소규모 카탈로그 파일의 그림자 (또는 마스터 병합). 마스터 병합이 완료 된 후에 더 작은 카탈로그 파일이 삭제 됩니다. 삭제 된 카탈로그 파일을 저장 하는 블록을 0으로 과정이 있습니다.

## <a name="continuous-replication"></a>연속 복제
연속 복제 (로그 전달 및 재생 라고도 함)는 Exchange online 만들고 고가용성, 사이트 복구 및 재해 복구를 제공 하려면 모든 사서함 데이터베이스의 복사본을 유지 관리 하는 기술입니다. 하나 이상의 사서함 데이터베이스 복사본의 비동기 업데이트를 수행 하는 기술을 제공 하려면 Exchange Server 데이터베이스 크래시 복구 지원 기술을 활용 하는 연속 복제 합니다. 각 사서함 서버 1MB 트랜잭션 로그 파일의 순차적 집합에 있는 로그 레코드와 활성 데이터베이스 (예: 사용자 전자 메일 활동)에서 데이터베이스 업데이트를 기록 합니다. 이 파일 집합이 로그 스트림 라고 합니다. 연속 복제에서 로그 스트림 비동기적으로 데이터베이스의 복사본을 하나 이상의 업데이트 하려면도 사용 됩니다. 이 로그를 전송 하 여 작업을 수행 현재 데이터베이스의 수동 복사본을 포함 하 고 수동 데이터베이스 복사본에 재생 하는 해당 하는 위치에 있습니다. 현재 데이터베이스에서 모든 로그는 데이터베이스의 수동 복사본에 대해 재생 하는 경우 두 데이터베이스는, 해당 하는 다음이 고 활성 데이터베이스에 대 한 실제 변경 해당 데이터베이스의 모든 수동 복사본에 복제 되는이 프로세스를 통해은 합니다.

모든 삭제 된 사서함 데이터베이스에서 사서함 항목 또는 전체 사서함, 여부 및 일시 삭제 또는 하드-삭제 등의 여부를 실제에 변경을 내용을 나타냅니다 활성 데이터베이스입니다. 페이지 비우기도 현재 데이터베이스에 대 한 실제 변경 특성을 갖습니다. 이러한 변경 내용이 연속 복제 이라는 프로세스를 통해 로그 파일에 기록 하 고 데이터베이스의 수동 복사본에 대해 이러한 로그 파일이 재생, 이러한 수동 데이터베이스에는 동일한 실제 변경 사항이 발생 합니다.
